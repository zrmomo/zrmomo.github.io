  • 二分查找的原理是什么

    1. 首先，需要有一个已排序的数组。如果数组未排序，可以先使用排序算法进行排序

    2. 在数组中找到中间元素。如果中间元素正好是要查找的元素，则搜索过程结束

    3. 如果要查找的元素大于中间元素，则在数组大于中间元素的那一半中查找；如果要查找的元素小于中间元素，则在数组小于中间元素的那一半中查找

    4. 重复步骤2和3，直到找到要查找的元素，或者搜索范围被缩小为0

      二分查找的时间复杂度是O(logn)，其中n是数组的元素数量。这是因为每次查找都会将搜索范围缩小一半。但是，二分查找的缺点是它需要一个已排序的数组，如果数组未排序，那么排序的时间复杂度可能会高于O(logn)。此外，二分查找只适用于顺序存储结构，如数组

  • 快排的复杂度是什么

    平均时间复杂度为O(nlogn)，在最好情况下时间复杂度为O(nlogn)，而在最坏情况下时间复杂度为O(n^2)。

  • 快排的原理或过程了解吗

    1. 选择一个基准元素（pivot），通常选择数组中的第一个元素或随机选择一个元素。
    2. 将数组中的元素按照与基准元素的大小关系进行划分，将小于基准元素的值放在基准元素的左边，将大于基准元素的值放在基准元素的右边。这个过程称为分区（partition）。
    3. 对基准元素左边和右边的子数组分别递归地进行快速排序。
    4. 当递归到子数组的大小为1或0时，排序完成。

  • 网络相关的优化

  1. 使用高效的数据格式：
    • 选择合适的数据格式，如JSON或者Protocol Buffers，这些格式可以显著减少数据的大小和解析时间。
  2. 数据压缩：
    • 在服务器端开启GZIP或Brotli等压缩方法，可以减少传输的数据量，加快加载速度。
  3. 缓存策略：
    • 利用HTTP缓存头（例如Cache-Control, ETag, Last-Modified）合理缓存数据，减少不必要的网络请求。
  4. 网络请求优化：
    • 使用批量请求减少HTTP的请求数量，或者使用WebSockets进行实时通信。
    • 对于移动设备，可以使用OkHttp等库来调整并发请求和连接池的大小，优化TCP/HTTP的性能。
  5. 图片和视频优化：
    • 根据客户端分辨率和需求对图片和视频进行适当的压缩和格式转换（如WebP格式的图片）。
  6. 懒加载（Lazy Loading）：
    • 只加载用户可视区域内的内容，而其他内容则在滚动到可视区域时加载。
  7. 预加载（Preloading）：
    • 在用户可能需要之前，事先加载资源。
  8. 优先级加载：
    • 确保关键资源（比如CSS文件）首先被加载，以提高用户感知的加载速度。
  9. 减少服务器响应时间：
    • 使用CDN分发内容，使资源尽可能靠近用户。
    • 优化服务器配置和数据库查询，以减少响应时间。
  10. 连接保持活动（Keep-Alive）：
    • 使HTTP连接保持开启状态，以便多个请求可以复用相同的TCP连接，减少TCP握手时间。
  11. 分页或分块获取大数据：
    • 当需要加载大量数据时，通过分页或者分块的方式逐步加载，减少单次加载的数据量。
  12. 错误和超时处理：
    • 合理设置超时时间，对请求失败进行重试策略和错误处理，能够改善不稳定网络环境下的用户体验。
  13. 移动网络适配：
    • 检测网络类型（WiFi、4G等），对不同的网络条件进行适应，例如在WiFi下加载高质量资源，在移动网络下加载低质量资源。
  14. Web请求的并发限制：
    • 在并发环境下，限制同时发出的HTTP请求数量，避免并发请求带来的服务器压力和客户端性能问题。

  • 如何实现HTTPS建连，回传像HTTP一样快（连接复用没有生效时怎么做到呢，能不能通过空间换时间实现）

  • 有没有做过弱网优化

优化重试机制：智能化的重试策略，能在网络恢复时自动尝试重新连接，而不是简单地在请求失败后立即重试。简单说就是再当前场景下，网络恢复后重新请求网络。

  • 熟悉垃圾回收机制，你简单介绍下？

    垃圾回收是一种自动管理内存的机制，它负责检测和释放不再被引用的对象，以防止内存泄漏和浪费。在 Android 中，主要使用的垃圾回收机制是标记-清除（Mark and Sweep）和分代回收。

  • 标记一定会被回收吗？不被标记可以不被回收吗？

    是的，标记的对象会被回收，不被标记的对象将不会被回收。垃圾回收器通过标记存活的对象来确定哪些对象可以被回收，不被标记的对象将保留在内存中。

  • 假如内存不够用，有些没被标记的会不会被回收呢？

    是的，如果内存不足，垃圾回收器可能会采取措施回收未被标记的对象，以释放内存。这通

常发生在极端情况下，如系统内存不足时。

  • 可达性分析算法在工作中有用过吗？

    可达性分析算法通常是垃圾回收器内部使用的一种算法，用于确定哪些对象可以被回收。开发人员通常不需要直接使用可达性分析算法，而是依赖于垃圾回收器来管理内存。

  • 生活中有什么是观察者模式这种现象？

  • 除了观察者，还有什么方案可以取消？通知？

  • 放服务器方案，启动下载之后问题，下载SO有流量、IO消耗，包瘦身和这些指标怎么权衡的？

    在选择将资源放在服务器端动态下载时，需要权衡以下因素：

    • 流量消耗：动态下载会消耗用户的数据流量，因此需要考虑用户的流量费用和体验。
    • IO消耗：服务器端资源动态下载可能需要较多的网络IO操作，因此需要考虑服务器的负载和性能。
    • 包大小：包瘦身可以减小应用的初始安装包大小，提高下载和安装速度。

    针对这些因素，需要综合考虑用户体验、服务器性能和网络成本，根据具体情况做出权衡决策。

  • 假设没网，SO，图片没下载下来？有什么解决方案？

    如果应用在没有网络连接的情况下需要使用动态下载的SO和图片，可以考虑以下解决方案：

    1. 预打包：将关键的SO文件和图片预打包到应用的安装包中，以确保即使没有网络也可以

    正常运行。
    2. 离线缓存：在有网络时下载所需的资源并进行离线缓存，以便在没有网络时使用已下载的资源。
    3. 优雅降级：设计应用以在没有网络时提供基本功能，不依赖于动态下载的资源。

    • 性能优化：确保默认图片方案不会导致性能下降，例如，加载占位图像的速度要快。
    • 用户体验：解释默认图片的用途，强调它是为了提供更快的加载速度，同时鼓励用户升级到高质量图片。
    • A/B 测试：进行A/B测试以验证默认图片方案是否对用户行为产生负面影响，根据测试结果做出调整。
    • 用户教育：向用户解释默认图片的用途，提醒他们下载高质量图片的好处。

  • 新包增加2M，比较什么内容你从什么方面入手呢？

    当新包增加2M时，可以考虑以下方面来确定哪些内容导致了包大小增加：

    1. 资源变化：检查新包中是否包含大量新增的资源文件，如图片、音频或视频。
    2. 库和依赖：查看新包是否引入了新的库或依赖项，它们可能会增加包大小。
    3. 代码变化：检查新代码是否增加了包大小，可能是因为新增的功能或模块。
    4. 资源压缩：确保资源文件进行了适当的压缩和优化。
    5. 代码混淆：使用代码混淆工具减小代码大小。
    6. 动态下载：检查是否在新版本中引入了动态下载资源的机制。

  • 多人团队开发中，怎么比较这个文件大小是谁增加的？

    在多人团队开发中，要比较特定文件大小是谁增加的，可以采用版本控制工具（如Git）的以下方法：

    1. 使用版本历史：查看文件的版本历史记录，可以查看每个版本中文件的大小变化。
    2. 作者信息：查看文件的作者信息，通常版本控制工具会记录每个提交的作者。
    3. 代码审查：在代码审查中讨论文件大小变化，团队成员可以提供相关信息。

  • 内存优化？基于什么背景做内存优化？

    内存优化是为了减少应用程序在运行时占用的内存，以提高性能和稳定性。内存优化的背景可能包括：

    1. 应用崩溃：应用因内存不足而崩溃或ANR。
    2. 性能问题：应用运行缓慢或卡顿，可能是因为内存过多被占用。
    3. 资源浪费：应用占用大量内存，导致设备性能下降和电池消耗增加。
    4. 用户体验：提高应用的响应速度和启动时间，以提供更好的用户体验。

  • 内存优化过程中用了什么方案，介绍一下？

    内存优化可能涉及多个方面，以下是一些可能用到的方案：

    1. 内存分析工具：使用工具如Android Profiler或Memory Profiler来分析内存使用情况，找到内存泄漏和冗余对象。
    2. 资源释放：及时释放不再使用的对象，特别是大对象和长生命周期对象。
    3. 使用弱引用：对于不需要强引用的对象，可以使用弱引用或软引用，以便垃圾回收器可以更容易地回收它们。
    4. 内存泄漏检测：使用工具如LeakCanary来检测内存泄漏并生成报告。
    5. 内存优化库：使用第三方库如LruCache来管理内存中的数据，以减少内存占用。
    6. 性能优化：通过减少不必要的计算和IO操作，减少内存消耗。
    7. 动态加载和释放资源：根据需要动态加载和释放资源，以减小应用的初始内存占

  • 有没有做过线程管控？

    线程管控通常是指管理和控制多线程应用程序的执行和资源分配。这可以包括线程池的使用、线程同步、线程调度和资源争用的处理。做过线程管控意味着熟悉多线程编程，以确保应用程序的稳定性和性能。

  • 如果一个第三方的SDK，在for循环里面疯狂创建对象，你怎么办？

    如果第三方SDK在for循环中频繁创建对象，可能会导致性能问题和内存占用增加。应对这种情况可以采取以下措施：

    1. 对象重用：尽量重用对象，避免重复创建和销毁。
    2. 批量操作：如果可能，尽量一次性批量操作对象，而不是在循环中逐个操作。
    3. 线程池：如果多线程环境下，考虑使用线程池来管理对象的创建和销毁，以避免过多线程创建对象。
    4. SDK升级：联系SDK提供商，查看是否有性能优化的新版本可用。

  • 线程池，你怎么防止别人不去new，而来用你的线程池呢？

    防止别人不去新建线程，而使用你的线程池可以考虑以下方法：

    1. 文档和规范：明确文档和规范中的线程池使用规则，强调应该使用线程池而不是手动创建线程。
    2. 封装：将线程池封装为一个私有成员，并不暴露线程池的创建和管理方法。
    3. 访问控制：使用访问控制修饰符，将线程池设置为私有或受保护，限制外部访问。
    4. 代码审查：在代码审查中检查是否有违反线程池使用规则的代码。
    5. 教育和培训：对团队成员进行培训，教育他们正确使用线程池的重要性。

    避免新团队成员在不了解线程池的情况下手动创建线程，可以考虑以下方法：

  • 模块化组件化？说说你理解的组件化？

    模块化和组件化是一种软件架构设计的方式，用于将复杂的应用拆分成可重用的部分，以提高代码的可维护性和可扩展性。

    • 模块化：模块化是将应用拆分成相对独立的模块或功能单元，每个模块负责特定功能或特性。这些模块可以是代码模块、库、插件等，它们之间通过明确定义的接口或API进行通信和交互。模块化的好处包括代码分离、单一职责、可重用性和易于测试。
    • 组件化：组件化是在模块化基础上更进一步，将应用拆分成多个独立的组件，每个组件可以独立开发、构建和部署。组件可以包括不同的应用页面、功能、服务等，它们可以独立升级和替换。组件化的好处包括更高的团队协作、更快的开发迭代和更好的模块隔离。

  • 组件化通信有哪几种方式？

    在组件化架构中，不同组件之间需要进行通信，常见的通信方式包括：

    1. 隐式意图（Intent）：通过发送隐式意图，启动目标组件的活动或服务。
    2. 显式意图（Explicit Intent）：明确指定目标组件的类名来启动。
    3. 本地广播：使用本地广播来在同一应用内的组件之间进行通信。
    4. 事件总线：使用事件总线库（如EventBus）来发布和订阅事件，实现组件间的松耦合通信。

  • 快排的原理讲一下？：快速排序（Quicksort）是一种基于分治思想的高效排序算法。其原理如下：

    1. 选择一个元素作为基准（通常是数组中的第一个元素）。
    2. 将数组分成两个子数组：小于基准的元素和大于基准的元素。
    3. 对这两个子数组递归地应用快速排序。
    4. 最后，将两个排序好的子数组和基准合并在一起，得到最终的有序数组。

Activity启动模式适用场景

  1. standard（标准模式）：
    • 场景：适用于大多数情况。每次启动 Activity 都会创建一个新的实例，并放入任务栈的顶部。
    • 示例：浏览器、新闻阅读等常规应用中的 Activity。
  2. singleTop（单顶模式）：
    • 场景：当 Activity 已经位于任务栈的顶部时，再次启动它时不会创建新的实例，而是调用其 onNewIntent() 方法。适用于接收通知或点击通知后跳转到应用的详情页等情况。
    • 示例：新闻详情页、聊天页面等详情页。
  3. singleTask（单任务模式）：
    • 场景：系统只会存在一个该 Activity 的实例，如果要启动的 Activity 已经存在于任务栈中，系统会将它上面的所有 Activity 出栈，使得该 Activity 成为任务栈的根。
    • 示例：应用的主界面、主页、欢迎页等只需要一个实例的界面。
  4. singleInstance（单实例模式）：
    • 场景：系统会为该 Activity 创建一个新的任务栈，且该任务栈只包含该 Activity 的实例，其他 Activity 无法进入该任务栈。适用于需要独立运行的 Activity。
    • 示例：应用的播放器页面、地图导航页面等单独运行的功能界面。
