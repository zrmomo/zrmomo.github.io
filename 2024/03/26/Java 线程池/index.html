线程池

发表于2024-03-26|更新于2024-03-14
|阅读量:

简介

​ 重用存在的线程，减少对象创建、消亡的开销，提升性能。 可有效控制最大并发线程数，提高系统资源的使用率，同时避免过多资源竞争，避免堵塞。 提供定时执行、定期执行、单线程、并发数控制等功能。

  • int corePoolSize: 核心线程数，即使线程是空闲的，也会一直存活在线程池中，除非设置了allowCoreThreadTimeOut。

    allowCoreThreadTimeOut(boolean) 方法用于控制核心线程是否允许超时回收。

  • int maximumPoolSize: 线程池中允许的最大线程数。当工作队列满了，并且已创建的线程数小于最大线程数时，线程池会再创建新的线程执行任务。

  • long keepAliveTime: 当线程数大于核心线程数时，这是多余空闲线程在终止前等待新任务的最长时间。

  • TimeUnit unit: keepAliveTime参数的时间单位。

  • BlockingQueue<Runnable> workQueue: 用于保存等待执行的任务的队列。这个队列仅用于保存由execute方法提交的Runnable任务。

  • RejectedExecutionHandler handler: 当线程池已经达到最大容量并且工作队列也满了时，用于处理新提交的任务的拒绝策略。

使用

线程池提交任务submit和execute的区别？

  1. 返回值:
    • execute(Runnable command): 没有返回值。它接受一个Runnable对象作为参数。
    • submit(Callable<T> task)或submit(Runnable task): 返回一个Future<T>对象。对于submit(Runnable task)，如果任务正常完成，Future的get方法将返回null。
  2. 异常处理:
    • execute: 如果在执行任务时抛出异常，那么异常将被传递给未捕获异常处理器（如果设置了的话），否则异常将被忽略。
    • submit: 如果任务执行过程中抛出异常，异常将被封装在返回的Future对象中，可以通过调用Future的get方法来获取异常。
  3. 任务类型:
    • execute只能接受Runnable类型的任务。
    • submit可以接受Runnable或Callable类型的任务。Callable任务允许有返回值，并且可以抛出经过检查的异常。

​ 通常情况下，如果需要处理异步任务的结果或异常，使用submit比较方便，因为它返回一个Future对象，可以通过这个对象来获取任务的结果或者检查任务是否已经完成。如果不需要关心任务的结果，只是想简单地执行一个任务，那么使用execute就足够了。

线程池执行流程

将任务添加

  • 当线程池中的线程数小于核心线程数时，新提交的任务将创建一个新的核心线程来执行该任务，即使此时线程池中存在空闲线程。
  • 当线程池中的线程数达到核心线程数时，新提交的任务将被放入任务队列中，等待线程池中的核心线程调度执行。
  • 当任务队列已满且线程池中的线程数量小于最大线程数时，新提交的任务会创建非核心线程执行任务。
  • 当提交任务数超过最大线程数时，新提交的任务将由拒绝策略处理。
  • 当线程池中的线程数超过核心线程数时（即非核心线程），如果这些线程的空闲时间达到了keepAliveTime，这些空闲线程将被关闭。
  • 当设置allowCoreThreadTimeOut(true)时，线程池中的核心线程在空闲时间达到keepAliveTime后也将被关闭。

线程池创建

Executors工具类中创建线程池一共有6种，每种有一个重载方法，所以能创建线程池方法一共12个。

    • 创建一个固定大小的线程池，线程数和核心线程的值都是传入的值，所有提交的任务都会在这些线程中执行。

  2. newCachedThreadPool

    • 创建一个可缓存的线程池，如果线程池的大小超过了处理需求，将回收空闲线程，否则添加新线程。

  3. newSingleThreadExecutor

    • 创建一个单线程的Executor，以无界队列方式来运行该线程。

  4. newScheduledThreadPool

    • 创建一个固定大小的线程池，支持定时及周期性任务执行。

  5. newSingleThreadScheduledExecutor

    • 创建一个单线程的定时任务线程池，可以用于延迟执行任务或者定期执行任务。

  6. newWorkStealingPool

    • 创建一个工作窃取算法的线程池，该线程池适用于大量短暂异步任务的场景。

代码解析

newFixedThreadPool：

​ 创建的线程池的线程数和核心线程的值都是传入的值，因为没有非核心线程所以存活时间设置没有意义。且任务队列为LinkedBlockingQueue默认是无限制的，所以可以一直往任务队列里加任务，永远无法创建非核心线程。进行任务的线程一直都是那么几个核心线程。

​ 核心数与最大数设置为同一个值，以此来保证并发控制。但由于这种特殊无界的特性，需要关注到是否合理使用防止OOM的发生。

优点：

  • 控制资源使用

    固定数量的线程池有效控制了系统资源的使用。避免系统资源（如内存）被大量线程耗尽。

缺点：

  • 灵活性差

    不能动态调整，在某些瞬时高并发的场景下，有界线程池可能无法及时响应大量的任务，如果设置的线程数较小，并且任务提交速度较快，可能导致部分任务一直等待执行，产生线程饥饿的情况。

总结：

​ 有界线程池适用于相对稳定的工作负载，能够限制线程数量，防止资源耗尽。在选择线程池类型时，需要根据应用场景和性能需求来进行权衡。

newCachedThreadPool

  核心线程数为0，线程池最大线程数为无限制，线程空闲时间为60秒，任务队列为SynchronousQueue一个不存储元素的队列。

​ 这中线程池每个提交的任务都会创建一个非核心线程去执行，提交100个任务就会创建100个线程。但是线程的存活时间有要求。

优点：

​ 无界线程池会根据需求动态地创建新线程，无上限地适应任务的数量。使得线程池能够灵活地适应工作负载的变化。

缺点：

​ 线程数量没有上限

总结：

​ 无界线程池适用于任务短暂、处理速度快的场景，

newSingleThreadExecutor

​ 这种线程池适合执行那些有顺序要求的任务，且减少了线程的创建和销毁开销，如果任务抛出异常而导致线程终止，会创建一个新的线程来取代原来的线程。

与newFixedThreadPool的区别

FinalizableDelegatedExecutorService类里面多了一个finalize()方法，该方法实际上调用的是线程池的shutdown()方法把线程池状态变为SHUTDOWN（即把任务执行完而不接受新任务）。

而二者之间主要区别在于垃圾回收时Single线程池多了一步销毁方法，其次是Fix线程池后期可以更改线程池数量，Single不可以（因为本来就是单一）

newScheduledThreadPool

使用

源码

​ 这种线程池用于执行延迟任务或者定时任务。核心线程数为传入的值，最大线程数为无限制，线程空闲时间由DEFAULT_KEEPALIVE_MILLIS确定，任务队列为DelayedWorkQueue，这是一种按照超时时间排序的队列结构

优点：

  • 定时执行和周期性执行

    ScheduledThreadPoolExecutor可以精确地控制任务的执行时间和频率，适用于需要定时或周期性执行任务的场景。

  • 资源使用相对稳定

    由于有核心线程数的限制，即使在任务数量较多的情况下，也能保持资源使用的稳定性。

缺点：

  • 可能导致资源耗尽

    虽然有核心线程数的限制，但最大线程数为无限制，如果提交的延迟任务数量过多，而且执行时间较长，可能会创建大量线程，导致资源耗尽。

  • 延迟任务堆积

    如果任务提交速度远大于处理速度，可能会导致延迟任务在队列中堆积，从而影响到任务的准时执行
